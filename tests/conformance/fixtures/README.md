# Fixture corpus

A pinned corpus of documents for the parse-conformance suite (see
`tests/conformance/README.md`), committed as raw bytes with a sha256-verified
manifest (`manifest.json`, generated by `generate-manifest.mjs`).

## Regenerating the manifest

After adding, removing, or editing any fixture file:

```sh
node tests/conformance/fixtures/generate-manifest.mjs
```

(or `npm run test:conformance:manifest` from the repo root). This recomputes
every fixture's sha256 and byte count from what's actually on disk and
rewrites `manifest.json`. `harness/fixtures.js`'s `loadFixture` verifies a
fixture's sha256 against the manifest on every load and throws loudly on a
mismatch — either the fixture was edited without regenerating the manifest,
or the manifest is stale.

Every fixture's `source`/`license` in `manifest.json` comes from
`FIXTURE_META` in `generate-manifest.mjs`, written by hand by whoever added
the file — never inferred from the path or content. A manifest entry is a
provenance claim; make it true.

## Categories

`municipal`, `legal`, `news`, `literary`, `ocr-damaged`, `adversarial`,
`degenerate` — see each fixture's `notes` field in `manifest.json` for what
specifically it's built to exercise. `literary-frankenstein` is not
duplicated here; it references the existing vendored copy at
`tests/fixtures/frankenstein.txt`.

## Regenerating a larger `huge-single-line` instance

`degenerate/huge-single-line.txt` is a single unbroken line with no
newlines — deliberately small (see its manifest `notes`) so it transcribes
reliably through this repo's tooling. To stress-test with a much larger
instance locally (the spec's own suggested scale is ~8MB), generate one
directly rather than committing it:

```sh
node -e "
  const unit = 'The committee reviewed the matter and Delgado moved to approve the item without further discussion while Reyes abstained';
  const n = 65000; // ~8MB
  const text = Array(n).fill(unit).join(' ').replace(/ abstained\$/, '.');
  process.stdout.write(text);
" > /tmp/huge-single-line-8mb.txt
```

and point a one-off `readWithSeed(fs.readFileSync(...))` call at it — this is
a local stress test, not something to commit to the pinned corpus.
